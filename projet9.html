L'idée du projet

Jusqu'où une IA agentique peut-elle construire seule un vrai rapport Power BI — pas juste écrire du code, mais monter le modèle de données, la mise en page, le thème, et vérifier elle-même le rendu dans Power BI Desktop ?

J'ai piloté Claude (l'agent IA d'Anthropic, via Claude Code) sur cette tâche complète, en gardant le rôle de superviseur : je donne le brief, je contrôle, je débloque quand il faut une action manuelle dans l'interface Windows. Le reste — structure, choix techniques, débogage — est laissé à l'agent.

Je m'y suis pris en deux temps : un premier projet « bateau » pour apprendre l'outillage et ses pièges, puis un vrai projet reproduisant un exercice de formation Power BI.

Projet 1 · rodage

Un dashboard de test

Brief volontairement simple : un dashboard de ventes avec un jeu de données fictif (144 transactions, 4 régions, 3 catégories). Objectif réel : découvrir la chaîne d'outils, rencontrer les erreurs, et surtout faire écrire par Claude un guide de tout ce qu'il a appris — pour cadrer la session suivante.

Projet 2 · le vrai

Suivi Réel vs Budget

Reproduire fidèlement une matrice de pilotage budgétaire vue dans une vidéo de formation Power BI : Réel vs Budget par service, décomposé jusqu'à la nature de charge, avec alerte colorée et un commentaire qui explique automatiquement les dépassements. Aucune donnée réelle : tout est généré en Python.

Projet 1 — le dashboard de test généré par l'agent (capture Power BI Desktop).

La stack & les dépôts utilisés

Le principe : ne jamais deviner le format des fichiers Power BI, mais s'appuyer sur l'outillage officiel Microsoft pour guider, valider et vérifier chaque étape.

Ce que je retiens sur le choix d'outils : le plus fiable est de partir des skills officielles Microsoft (skills-for-fabric), qui encadrent tout le travail et cadrent l'agent, puis d'y brancher le MCP adapté à la couche visée (modèle vs rapport). C'est ce cadre officiel qui évite la plupart des erreurs, plutôt qu'un outil communautaire seul.

Les skills officielles Microsoft — github.com/microsoft/skills-for-fabric

Un dépôt Microsoft (MIT) de « skills » : des instructions markdown que l'agent lit pour savoir manipuler Fabric. Il couvre toute la plateforme ; voici les skills Power BI qui servent ici, chacune avec un rôle précis :

  • powerbi-report-authoring — écrire/éditer le rapport PBIR (pages, visuels, filtres, slicers, thèmes, formatage) et la boucle valider → recharger → capturer. C'est la skill centrale de ce projet.
  • powerbi-report-design — le design ouvert : thème, style visuel, layout, archétypes de dashboard, critique de rendu.
  • powerbi-report-planning — le cadrage amont : transformer un besoin en spécification de rapport avant de construire.
  • powerbi-report-management — la gestion côté Fabric (créer/lister/supprimer des rapports dans un workspace).
  • semantic-model-authoring — construire le modèle : tables, relations, mesures DAX, TMDL.
  • semantic-model-consumption — interroger un modèle existant (requêtes DAX, exploration).

Ces skills s'appuient sur deux CLI officielles Microsoft : @microsoft/powerbi-report-authoring-cli (valider le PBIR, connaître les propriétés exactes des visuels) et @microsoft/powerbi-desktop-bridge-cli (piloter Power BI Desktop : ouvrir, recharger, capturer le rendu réel).

Les deux MCP : officiel (modèle) vs communautaire (rapport)

Un MCP est un serveur qui expose des outils actionnables par l'agent. Deux existent, sur deux couches différentes :

Officiel · Microsoft

powerbi-modeling-mcp

Le MCP officiel Microsoft. Il agit sur le modèle sémantique : tables, colonnes, relations, mesures DAX, rôles de sécurité, hiérarchies, exécution/validation de requêtes DAX. Mais il ne touche pas au rapport — sa doc précise qu'il « ne peut pas modifier les pages de rapport ».

Communautaire

powerbi-report-builder

Un MCP tiers (auteur Jonathan, MIT) qui embarque powerbi-report-mcp. Il expose ~30 outils pbir_* pour construire le rapport (pages, visuels, thème, layout) sans écrire le JSON à la main — la couche que le MCP officiel ne couvre pas.

En clair : pour la partie modèle (DAX/TMDL), il existe un MCP officiel Microsoft ; pour la partie rapport (PBIR), seul un MCP communautaire répond aujourd'hui. Dans ce projet, le modèle a été écrit à la main en TMDL/DAX et le rapport assemblé via le MCP communautaire.

Le reste de la stack

  • Power BI Desktop — installé à la volée via winget (paquet MSIX du Microsoft Store), piloté par le Desktop Bridge pour vérifier le rendu réel.
  • Python — pandas / openpyxl pour générer de zéro les jeux de données fictifs (jusqu'au format brut « pivoté » à retraiter dans Power Query).
  • Formats — projet PBIP (dossier versionnable), rapport PBIR, modèle TMDL, mesures DAX, transformations Power Query M.

Ce que j'ai demandé à Claude

Le brief est resté volontairement minimal, pour voir ce que l'agent décide seul :

  • « Construis le rapport, ne te contente pas de proposer du code » — installer les outils, écrire les fichiers, valider, ouvrir Desktop, se relire.
  • « Vérifie toi-même le résultat » — recharger le rapport et prendre une capture d'écran après chaque changement, plutôt que de me laisser deviner si c'est bon.
  • « Reproduis exactement ce que montre la vidéo » pour le projet 2 — même hiérarchie, mêmes règles de couleur, même colonne de commentaire.
  • « Transforme tes erreurs en guide » — à la fin du projet 1, écrire un mode d'emploi réutilisable des pièges rencontrés (disponible en téléchargement plus bas).

Ce guide écrit par l'agent lui-même est ce qui a rendu le projet 2 beaucoup plus fluide : la session suivante démarrait déjà en connaissant les bons réflexes.

Le vrai projet : Suivi Réel vs Budget

Un script Python génère toutes les sources : 9 services, 13 natures de charge, environ 3 800 écritures comptables sur 17 mois, avec pour chaque écriture un fournisseur réaliste — et un fournisseur volontairement mis en pic certains mois, pour faire « vivre » la colonne de commentaire.

6tables (2 de faits, 3 dimensions, 1 calendrier)
8mesures DAX
~3 800écritures générées en Python
3niveaux de hiérarchie

Le modèle de données

Deux tables de faits (Comptabilite au jour le jour, Budget au mois) partagent trois dimensions (Ref_Services, Ref_Comptes, CALENDRIER). Le calendrier n'est pas importé mais calculé en DAX (CALENDARAUTO() + colonnes ANNEE-MOIS, TRIMESTRE, SEMAINE, année fiscale), conformément à la consigne de la formation.

Le modèle en flocon : deux tables de faits (Comptabilite, Budget), trois dimensions (Ref_Services, Ref_Comptes, CALENDRIER) et la table _Mesures — vue Modèle de Power BI Desktop.

La partie la plus intéressante : le commentaire automatique

Une mesure DAX ne s'affiche qu'au niveau le plus fin (une nature de charge) et seulement au-delà d'un seuil de dépassement, en pointant le fournisseur responsable :

// ne se déclenche qu'au niveau Nature de charge, et au-delà de +0,2 %
Analyse des dérives =
    VAR EstAuNiveauNature = ISINSCOPE(Ref_Comptes[NATURE_CHARGE])
    VAR Ecart = [Écart %]
    RETURN
        IF(
            EstAuNiveauNature && NOT ISBLANK(Ecart) && Ecart > 0.002,
            "Dépassement " & FORMAT([Écart €], "+#,##0;-#,##0") &
            " € — concentré sur " & [Fournisseur Principal] &
            " (" & FORMAT([Part Fournisseur Principal], "0%") & " du réel)"
        )

Le visuel obtenu

Le rendu final de la page « Suivi Réel vs Budget » : hiérarchie Service → Charges de personnel / externes → nature de charge, avec pastille d'alerte, écart en € et %, et la colonne « Analyse des dérives » qui se remplit sur les dépassements.

Capture Power BI Desktop. La ligne en alerte rouge affiche le commentaire automatique : « Dépassement +45 € — concentré sur Lyreco (38 % du réel) ».

Le vrai point bloquant : le thème

La plupart des accrocs étaient mineurs et corrigés dans la foulée. Un seul a vraiment coûté du temps : les 4 cartes d'indicateurs (KPI) restaient visuellement vides — le cadre, la bordure et l'ombre s'affichaient, mais aucun chiffre ni libellé. Et pourtant : aucune erreur de validation, et la donnée était bonne (confirmée en changeant de type de visuel).

La cause : le fichier de thème (theme.json) contenait un sélecteur "$id": "default", copié depuis un exemple de la documentation. Cette syntaxe est valide dans un fichier de visuel, mais pas dans un fichier de thème : Power BI Desktop ignore la clé en silence et casse l'affichage du texte. Pire, le validateur officiel ne détecte pas l'erreur, car il ne vérifie pas la cohérence croisée thème ↔ visuel.

Le bug a été trouvé après plusieurs cycles test → capture → correction, avec l'aide de ma relecture attentive. La leçon retenue (et écrite dans le guide) : ne jamais mettre de sélecteur dans un fichier de thème, et pour un premier jet, préférer le thème officiel « Copilot » intégré à Power BI plutôt que d'en fabriquer un à la main.

Les autres accrocs (mineurs, corrigés automatiquement par Claude)

Aucun de ces points n'a bloqué le projet : ils ont été diagnostiqués puis corrigés par l'agent dans la foulée. Quelques exemples :

  • Un signe « $ » avalé par le terminal — un format monétaire dans un fichier TMDL devenait un octet invalide à cause d'un échappement bash. Fix : écrire les fichiers directement plutôt qu'en ligne de commande.
  • Confusion entre langage M et DAX — un suffixe numérique 54L, valide en DAX mais inexistant en Power Query M, faisait planter Desktop au chargement.
  • Schémas JSON devinés — plusieurs URLs de schéma (version.json, pages.json) fausses, retrouvées une par une via le validateur officiel.
  • Relation avec un ID de colonne invalide — les colonnes du calendrier étaient déclarées comme colonnes d'une table importée au lieu de vraies colonnes calculées ; redéclarées correctement.
  • Le slicer de date ne filtrait qu'un seul jour — créé en mode liste déroulante au lieu d'un curseur « entre deux dates » : le Réel s'effondrait pendant que le Budget restait entier.
  • Les totaux affichaient « 0 » au lieu de la pastille — mise en forme conditionnelle par icône réglée sur « valeurs seules » ; corrigée pour inclure les lignes de sous-total.

Ce que j'en retiens

  • Une IA agentique peut produire un livrable Power BI réel et se vérifier elle-même en pilotant Desktop — à condition d'être supervisée et débloquée sur les actions manuelles.
  • Le plus gros risque n'est pas le code mais les différences subtiles de format (thème vs visuel, M vs DAX) qu'aucun validateur ne signale.
  • Faire écrire à l'agent un guide de ses propres erreurs transforme un projet raté-puis-réussi en capital réutilisable : c'est une boucle de rétroaction appliquée à une IA plutôt qu'à une équipe.
